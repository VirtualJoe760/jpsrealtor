# `src/lib/db/migrations/` — drizzle-kit generated DDL (reference)

These files are produced by `npx drizzle-kit generate --config=drizzle.config.ts`
from the Drizzle schema barrel (`src/lib/db/schema/index.ts`). They are a
**generated reference / introspection artifact**, not the migration the platform
applies at tenant-provision time.

## The canonical per-tenant migration is `src/lib/reso/migrations/0001_init.sql`

`0001_init.sql` (owned by Agent 04, generated by `src/lib/reso/ddl.ts`) is the
single migration the provisioning runner (Agent 16) applies to each tenant's
fresh Neon database over the **direct** (non-pooled) connection. It includes the
parts drizzle-kit cannot emit:

- `CREATE EXTENSION postgis / pg_trgm`
- the PostGIS **GiST** index on `property.geom`
- **GIN(`jsonb_path_ops`)** indexes on every `extras` column
- the **pg_trgm GIN** index on `property.unparsed_address` (street resolve)
- the ported compound indexes (city/subdivision/status/type/beds/baths,
  cashflow expression indexes)

drizzle-kit emits btree indexes and column DDL only. The two are kept reconciled:
the Drizzle tables mirror `0001_init.sql`'s columns/types exactly (a reconciliation
test in `src/lib/reso/__tests__/ddl.test.ts` asserts the generator output is
byte-identical to the committed `0001_init.sql`).

**Bottom line:** treat the drizzle-kit output here as a typed view / diff aid.
Provisioning applies `0001_init.sql`. If you change the schema, update the
catalog (`src/lib/reso/data-dictionary.ts`) + regenerate `0001_init.sql`, then
re-run `drizzle-kit generate` to refresh this reference.
